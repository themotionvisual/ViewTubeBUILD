What I’d do for Channelytics + Research Lab + data tables + Oracle algorithm:
	1	Create one canonical analytics model
	•	Single normalized schema for video, channel, snapshot, experiment, insight.
	•	Both Channelytics and Research Lab read/write through this model only.
	2	Add a metric registry layer
	•	Define metrics once (ctr, avd, retention_drop_30s, velocity_24h, etc.).
	•	Each metric has formula, required fields, fallback logic, and display format.
	•	Removes duplicated calculations across views.
	3	Build a shared query/aggregation engine
	•	Centralized filters, grouping, ranking, and time-window logic.
	•	Power both table screens from one engine with different presets.
	4	Make “Oracle” a scoring pipeline, not a monolith
	•	Split into stages: feature extraction -> weighted scoring -> recommendation generation -> confidence.
	•	Store intermediate outputs so you can debug why a recommendation was produced.
	5	Unify table infra
	•	One reusable data table component with:
	•	server/client pagination
	•	virtual rows
	•	column config + saved views
	•	CSV export
	•	Channelytics/Research Lab become configuration, not separate table implementations.
	6	Introduce caching + sync strategy
	•	Stale-while-revalidate for expensive analytics queries.
	•	Background sync jobs for YouTube/API pulls.
	•	Snapshot versioning so Oracle compares apples-to-apples.
	7	Add validation + drift guards
	•	Schema validation at ingest.
	•	Metric sanity checks (e.g., CTR bounds, watch time null handling).
	•	Alert when upstream API changes break assumptions.
	8	Close the loop with experiments
	•	Research Lab generates hypotheses.
	•	Channelytics tracks outcomes.
	•	Oracle updates weights from experiment results.
